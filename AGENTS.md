# AGENTS.md — Правила работы агентов над Multiplayer Alchemy

Этот файл — обязательный контракт для каждого агента-исполнителя. Модель работы: **1 агент = 1 фаза** из [PLAN.md](PLAN.md). Агент получает идентификатор фазы (P1, P2, P3, S1, S2, D1, D2, C1, C2, C3, F1), выполняет ТОЛЬКО её и завершает работу.

---

## 1. Контекст проекта (минимум, который нужно знать)

* **Что строим:** кооперативная мультиплеерная игра «Алхимия» — см. [SPEC.md, раздел 0 «Замысел игры»](SPEC.md).
* **Архитектура:** Authoritative Server, монорепо `shared/` + `server/` + `client/` — см. [SPEC.md, раздел 1](SPEC.md). Стек и порты зафиксированы в [PLAN.md, правило 0.4](PLAN.md) — не пересматривать.
* **Декомпозиция работ и зависимости фаз:** [PLAN.md, правило 0.5](PLAN.md) (mermaid-диаграмма). Не начинать фазу, если её зависимости не выполнены (признак выполнения — соответствующие файлы существуют и коммит фазы есть в истории git).

Больше ничего о проекте знать не требуется. Полная спецификация нужна только там, где описание фазы прямо ссылается на разделы SPEC.md.

---

## 2. Роль агента-исполнителя фазы

Ты — исполнитель ровно одной фазы. Твой рабочий цикл:

1. **Прочитай** свою фазу в [PLAN.md](PLAN.md) целиком: секции «Цель», «Читает», «Создаёт/изменяет», «Приёмка».
2. **Прочитай** только файлы из секции «Читает» своей фазы и разделы SPEC.md, на которые фаза ссылается. Глобальные правила [PLAN.md, раздел 0](PLAN.md) обязательны всегда.
3. **Выполни** работу строго в границах секции «Создаёт/изменяет».
4. **Проверь** результат командами из секции «Приёмка». Все проверки зелёные — фаза готова.
5. **Зафиксируй** результат в git и запушь (см. раздел 4 ниже).
6. **Отчитайся**: какая фаза, какие файлы созданы/изменены, результат приёмки, хеш коммита.

### Специализация по типам фаз

| Фазы | Роль | Ключевые правила |
| :--- | :--- | :--- |
| P1–P3 | Инженер каркаса и контента | Точно следовать спискам файлов фазы; версии зависимостей ставить актуальные через npm, не выдумывать. |
| S1–S2 | Backend-инженер | Контракт событий — ТОЛЬКО из `shared/src/types.ts`; поведение — по [SPEC.md, разделы 3–4](SPEC.md). Smoke-тесты фазы обязательны. |
| D1–D2 | Дизайн-оператор Stitch | Работает ТОЛЬКО инструментами MCP `stitch` и записью в `design/`. Кодовую базу не читает и не изменяет. См. раздел 3 ниже. |
| C1–C3 | Frontend-инженер | Визуал — ТОЛЬКО из `design/` (выгрузки Stitch); события — только из `shared/src/types.ts`. Ничего не изобретать в дизайне. |
| F1 | Интегратор/QA | Единственная роль с правом читать весь проект. Правки — точечные, по найденным дефектам, с обоснованием. |

## 3. Жёсткие запреты (нарушение = провал фазы)

1. **НЕ исследовать проект целиком.** Никаких «осмотрюсь в кодовой базе»: читать разрешено только файлы из секции «Читает» твоей фазы. Всё, что нужно знать, уже там или в указанных разделах SPEC.md. Исключение — фаза F1.
2. **НЕ трогать чужие файлы.** Создавать/изменять можно только файлы из секции «Создаёт/изменяет». Матрица владения — [PLAN.md, «Сводная матрица владения файлами»](PLAN.md). Файлы с пометкой `[extends]` — только дополнять, существующий код не переписывать.
3. **НЕ генерировать дизайн самостоятельно.** Весь визуал — через MCP-сервер `stitch` с ОБЯЗАТЕЛЬНЫМ ожиданием ответа (опрос статуса, лимиты) — полный регламент в [PLAN.md, правило 0.2](PLAN.md). Stitch недоступен или не ответил за лимит — остановись и сообщи пользователю. Fallback на собственный дизайн ЗАПРЕЩЁН в любой формулировке («временно», «черновик», «placeholder-стили» — тоже запрещены; допустимы только нестилизованные заглушки, явно предусмотренные фазой).
4. **НЕ менять контракты.** `shared/src/types.ts` и `shared/src/constants.ts` после фазы P2 неприкосновенны. Обнаружил ошибку в контракте — останови фазу и сообщи, не чини молча.
5. **НЕ расширять объём.** Никаких «заодно отрефакторю», «добавлю полезную утилиту», «улучшу соседний файл». Non-goals проекта — [SPEC.md, раздел 0, «Не входит в объём»](SPEC.md).
6. **НЕ пропускать приёмку.** Фаза без зелёной приёмки не коммитится как выполненная.
7. **НЕ коммитить секреты.** `.cursor/mcp.json` содержит API-ключ и уже в `.gitignore` — не снимать это исключение, не вписывать ключи в код или документы.

## 4. Git-регламент

**Удалённый репозиторий:** `https://github.com/4il228/Multiplayer-Alchemy` (существует, пустой). Ветка — `main`, работаем в ней без feature-веток (фазы последовательны и не пересекаются по файлам, конфликты исключены планом).

### 4.1 Разовая инициализация (выполняет агент фазы P1 до начала работ)

```powershell
git init
git branch -M main
git remote add origin https://github.com/4il228/Multiplayer-Alchemy.git
git add SPEC.md PLAN.md AGENTS.md .gitignore
git commit -m "docs: project spec, plan and agent rules"
git push -u origin main
```

Перед этим убедиться, что `.gitignore` содержит `.cursor/mcp.json` (защита API-ключа).

### 4.2 Цикл каждой фазы

1. **Перед началом:** `git pull origin main` и `git status` — рабочее дерево должно быть чистым. Грязное дерево или неизвестные файлы — стоп, сообщить пользователю.
2. **Коммит — один на фазу**, только после зелёной приёмки. В коммит включать только файлы своей фазы (`git add <конкретные пути>`, НЕ `git add .`).
3. **Формат сообщения:** `<type>(<фаза>): <суть>` где type — `feat` / `chore` / `docs` / `fix` / `test` / `design`. Примеры:
   * `chore(P1): monorepo skeleton with shared/server/client workspaces`
   * `feat(S2): board handlers, locks and crafting logic`
   * `design(D1): lobby screen generated via Stitch MCP`
   * `fix(F1): cursor desync on reconnect`
4. **Пуш сразу после коммита:** `git push origin main`. Пуш не прошёл (отказ доступа, non-fast-forward) — не решать самостоятельно силовыми методами: НИКАКИХ `push --force`, `reset --hard`, правок `git config`. Сообщить пользователю.
5. Фаза F1 может дать несколько коммитов (`fix(F1): ...` на каждый дефект) плюс финальный `docs(F1): README`.

### 4.3 Бинарные артефакты дизайна

`design/*.png` и `design/*.html` (фазы D1/D2) коммитятся в репозиторий — это источник правды для фаз C1–C3.

## 5. Протокол отчёта о завершении фазы

Завершая работу, агент обязан выдать отчёт:

```
Фаза: <ID> — <название из PLAN.md>
Статус: ВЫПОЛНЕНА | ОСТАНОВЛЕНА (причина)
Файлы: <список созданных/изменённых>
Приёмка: <команда — результат, по каждой проверке>
Git: <хеш коммита> — <сообщение>, запушено в origin/main
Отклонения от плана: нет | <что и почему>
```

Если фаза ОСТАНОВЛЕНА (недоступен Stitch, ошибка в контракте shared, грязный git, невыполненные зависимости) — в отчёте указать точную причину и что требуется от пользователя. Частично сделанную работу НЕ коммитить.
